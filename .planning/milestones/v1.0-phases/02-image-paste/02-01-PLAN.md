  <name>Task 1: Install dependencies, DB migration, and image CRUD module</name>
  <files>package.json, src/lib/server/db.ts, src/lib/server/images.ts, .env</files>
  <action>
1. Install sharp and sortablejs:
   ```bash
   npm install sharp sortablejs
   npm install -D @types/sortablejs
   ```

2. Create `.env` file in project root with:
   ```
   BODY_SIZE_LIMIT=10M
   ```
   This overrides SvelteKit adapter-node's default 512KB body limit (Pitfall 1 from research). 10MB allows headroom above the 5MB per-image limit for FormData overhead.

3. Extend `src/lib/server/db.ts` with a version 2 migration. Re-read the `currentVersion` after existing migration and add:
   ```sql
   CREATE TABLE IF NOT EXISTS images (
     id INTEGER PRIMARY KEY AUTOINCREMENT,
     pad_id INTEGER NOT NULL REFERENCES pads(id) ON DELETE CASCADE,
     uuid TEXT NOT NULL UNIQUE,
     filename TEXT NOT NULL,
     mime_type TEXT NOT NULL DEFAULT 'image/webp',
     size_bytes INTEGER NOT NULL,
     sort_order INTEGER NOT NULL DEFAULT 0,
     created_at TEXT NOT NULL DEFAULT (datetime('now'))
   );
   CREATE INDEX IF NOT EXISTS idx_images_pad_id ON images(pad_id);
   CREATE INDEX IF NOT EXISTS idx_images_uuid ON images(uuid);
   ```
   Set `user_version = 2` after migration. Follow the exact pattern of the existing version 1 migration block.

4. Create `src/lib/server/images.ts` following the established pattern from `pads.ts` (prepared statements, typed interfaces):

   Define `Image` interface:
   ```typescript
   export interface Image {
     id: number;
     pad_id: number;
     uuid: string;
     filename: string;
     mime_type: string;
     size_bytes: number;
     sort_order: number;
     created_at: string;
   }
   ```

   Create prepared statements and export these functions:
   - `getImagesByPadId(padId: number): Image[]` -- ordered by sort_order ASC
   - `getImageByUuid(uuid: string): Image | undefined`
   - `insertImage(padId: number, uuid: string, filename: string, sizeBytes: number, sortOrder: number): Image` -- insert and return the created row
   - `deleteImage(uuid: string): boolean` -- delete row, return true if deleted
   - `updateImageOrder(orders: { uuid: string; sort_order: number }[]): void` -- update multiple rows in a transaction
   - `getPadImageTotalSize(padId: number): number` -- SUM(size_bytes) for quota check, return 0 if no images
   - `getNextSortOrder(padId: number): number` -- MAX(sort_order) + 1, or 0 if no images

   All queries must use prepared statements. The `updateImageOrder` function should use `db.transaction()` to batch updates.

<task type="auto">
  <name>Task 2: API endpoints for image upload, serve, delete, list, and reorder plus SSR load</name>
  <files>src/routes/api/pads/[...slug]/images/+server.ts, src/routes/api/pads/[...slug]/images/[imageId]/+server.ts, src/routes/api/pads/[...slug]/images/reorder/+server.ts, src/routes/[...slug]/+page.server.ts</files>
  <action>
1. Create `src/routes/api/pads/[...slug]/images/+server.ts` with POST and GET handlers:

   **POST handler (upload):**
   - Extract slug from params. Look up pad via `getPadBySlug(slug)`. If pad doesn't exist, create it via `createPad(slug)` (consistent with page load behavior). Need the pad's integer `id` for the FK.
   - Check `Content-Length` header against 5MB limit. If exceeded, return `error(413, 'Image too large (5MB max)')`.
   - Parse `request.formData()`, get the `image` field as `File`.
   - Validate it's an image: `file.type.startsWith('image/')`. If not, return 400.
   - Double-check `file.size` against 5MB limit (Content-Length can be spoofed).
   - Check per-pad quota: `getPadImageTotalSize(padId)`. If current total + file.size > 100MB, return `error(413, 'Pad image quota exceeded (100MB max)')`.
   - Read file into Buffer: `Buffer.from(await file.arrayBuffer())`.
   - Optimize with sharp:
     ```typescript
     const optimized = await sharp(buffer)
       .rotate() // Auto-apply EXIF orientation (Pitfall 6)
       .resize(2000, 2000, { fit: 'inside', withoutEnlargement: true })
       .webp({ quality: 80 })
       .toBuffer();
     ```
   - Generate UUID via `crypto.randomUUID()`, filename = `${uuid}.webp`.
   - Create upload directory: `data/uploads/{padId}/` using `mkdir({ recursive: true })`.
   - Write optimized buffer to `data/uploads/{padId}/{uuid}.webp`.
   - Get next sort_order via `getNextSortOrder(padId)`.
   - Insert into DB via `insertImage(padId, uuid, filename, optimized.length, sortOrder)`.
   - Return `json({ id: uuid, filename, size: optimized.length, sort_order: sortOrder })` with status 201.

   **GET handler (list):**
   - Look up pad by slug. If not found, return empty array `json([])`.
   - Return `json(getImagesByPadId(pad.id))`.

2. Create `src/routes/api/pads/[...slug]/images/[imageId]/+server.ts` with GET and DELETE:

   **GET handler (serve image file):**
   - Look up image by `params.imageId` (this is the UUID) via `getImageByUuid()`.
   - If not found, return 404.
   - Construct file path: `data/uploads/{image.pad_id}/{image.filename}`.
   - Read file with `readFile()`. If file missing on disk, return 404.
   - Return `new Response(data, { headers: { 'Content-Type': 'image/webp', 'Cache-Control': 'public, max-age=31536000, immutable', 'Content-Length': String(data.length) } })`.

   **DELETE handler:**
   - Look up image by UUID. If not found, return 404.
   - Delete filesystem file first: `unlink(path)`. Catch errors (file may already be gone).
   - Delete DB row via `deleteImage(uuid)`.
   - Return `json({ deleted: true })`.

3. Create `src/routes/api/pads/[...slug]/images/reorder/+server.ts` with PUT:

   **PUT handler:**
   - Accept JSON body: `{ orders: [{ uuid: string, sort_order: number }] }`.
   - Validate the array is present and non-empty.
   - Call `updateImageOrder(orders)`.
   - Return `json({ updated: true })`.

4. Update `src/routes/[...slug]/+page.server.ts` to load image metadata alongside pad content:
   - Import `getImagesByPadId` from `$lib/server/images`.
   - After getting the pad, fetch images: `const images = getImagesByPadId(pad.id)`.
   - Add `images` to the return object: `return { slug, content, version, images }`.
   - Image metadata is small enough to include in SSR load -- it never blocks text rendering. The actual image binaries are loaded lazily via separate GET requests.
